\subsection{The Agora Platform}\label{sec:agora}

Agora is an infrastructure platform that instantiates the MMATPG for real Lean~4 projects. It provides library mechanics (git-backed shared and private libraries with verified publication), market mechanics (collateralized target-conditioned contracts with automated settlement), and an agent interface (MCP-based tool access for LLM agents).

The Agora runtime consists of three backend services and an external orchestration harness.

\paragraph{Library Mechanics.}
The service responsible for all Lean interaction. It maintains an LRU cache of project repositories cloned from GitHub forks onto Agora-specific branches, managing all project initialization, building, and target extraction. Moreover, the library mechanics module invokes VerifiedAgora to validate or reject attempted contributions and merge them to the shared branch. A dependency-aware request scheduler serializes concurrent contributions to the same project, ensuring that non-ephemeral completions correctly unblock downstream module builds.

\paragraph{Market Mechanics.}
The financial core, owning wallets (cash balances, private assets, posted offers), target statuses, and the public offer book. It enforces a (i) money conservation invariant: the net asset diff of each transaction, after simplifying all target-conditioned assets against current resolution data, must have zero net coefficient; and a (ii) liquidity invariant: each wallet's minimum value (cash plus lower bounds of held assets) must remain non-negative. When a target is resolved, Market Mechanics records the proof time and proving agent, simplifies all referencing assets and offers, and converts any fully determined assets into wallet balance.

\paragraph{Backend Interface.}
The public API gateway coordinating the two mechanics services. It handles agent authentication (API keys with bcrypt hashing, or Supabase JWTs), synchronizes target metadata between library mechanics and market mechanics, broadcasts activity events over WebSocket (contributions, target updates, trades), and exposes a MCP server. The backend delegates all Lean validation to library mechanics and all financial invariant enforcement to market mechanics.

\paragraph{Agent Orchestrator.}
A separate Claude Agent SDK-based harness that provisions agent containers, manages their lifecycle, and provides a persistent loop for LLM interaction. Namely, each agent container runs an LLM with access to MCP-based library tools (e.g. list projects, submit contributions, etc.); market tools (e.g. create/take/cancel offers, transfer assets, etc.); Lean LSP tools (e.g. goal state, hover info, etc.); and standard filesystem tools.

The agent runtime maintains structured state (active project, target, wallet, build status, market positions), and includes recovery hints for common failure modes (edit mismatches, Lean compilation errors, path resolution issues). After each LLM response, the harness reengages with updated status, recent market events, and action suggestions. Auto-compaction summarizes long conversations into a memory file to prevent context overflow.


\subsubsection{Verification}\label{sec:verified-agora}

VerifiedAgora is a Lean~4 library that provides the trusted verification layer.

\paragraph{Target attribute.}
A Lean tag attribute \texttt{@[target]} marks declarations as Agora proof obligations. Contracts in the market are tied to these tagged declarations.

\paragraph{Axiom policy.}
Non-target declarations may only transitively depend on three foundational axioms: \texttt{propext}, \texttt{Quot.sound}, and \texttt{Classical.choice}. Target declarations may additionally depend on \texttt{sorryAx} while unresolved, but a \emph{solved} target must be sorry-free and axiom-clean. Axiom usage is collected via batched recursive traversal of declaration types and values, with cycle protection and cross-declaration caching.

\paragraph{Contribution validation.}
When an agent submits Lean content, VerifiedAgora: (i) compiles the submission; (ii) extracts and matches declarations against the existing target descriptor, checking type/kind/attribute/instance compatibility and axiom policy; (iii) produces an automerged file where valid solved targets replace sorry'd versions, invalid submissions revert to originals, and new non-target declarations are included; (iv) writes the automerged file and runs \texttt{lake build}; if the build fails, the original file is restored. This conservative pipeline ensures agents cannot break the repository.



\subsubsection{Inference-Time MMATPG Mapping}\label{sec:inference-mapping}

We now define how each MMATPG component is instantiated at inference time in an online fashion, when real LLM agents operate within the deployed Agora scaffold.

\paragraph{Time Discretization}\label{sec:time-discretization}

The MMATPG is a discrete-time game with simultaneous actions. In the real Lean setting, agents operate asynchronously in continuous wall-clock time: proof attempts take variable durations, market actions are near-instantaneous, and agents do not move in lockstep.

We discretize time into decision epochs indexed by $t \in \mathbb{N}$, where each epoch corresponds to one LLM interaction step (one prompt $\to$ response cycle). Within a single epoch, an agent may take exactly one high-level action: initiate a proof attempt, publish a result, submit a query, or take a market action. Proof attempts are durative: initiating a proof at epoch $t$ with budget $\tau$ means the strategic agent is occupied (forced to take $\bot$) for the next $\tau$ epochs while the prover subagent it iterates on the proof. Publication and market actions are instantaneous (consume one epoch).

This discretization has two consequences. First, the ``budget'' $\tau$ of a proof attempt is measured in LLM interaction steps, not wall-clock seconds or tokens. This normalizes across agents with different inference speeds. Second, because agents operate asynchronously, the simultaneous-action semantics of the MMATPG are approximated: in practice, noop's are synthetically inserted [\textbf{todo explain how/when/why?}]

\begin{remark}
This design choice reflects a tradeoff between fidelity and tractability. A fully asynchronous, continuous-time model would be more realistic but significantly more complex to analyze and implement. Additionally, the lack to distinction between faster, smaller models and slower, larger models is a limitation to the practical measurement of heterogeneous agent capabilities. Future work could explore more nuanced time models that account for these differences, such as token-based budgets or continuous frameworks.
\end{remark}

\paragraph{Component Mapping}
\begin{itemize}
    \item Formulas and sequents: Lean declarations of type \texttt{Prop} serve as formulas. Each \texttt{@[target]}-tagged declaration is a target formula. The sequent $[\Gamma \proves \phi]$ is identified with the target declaration, since the antecedent $\Gamma$ is determined by the import graph and the current shared branch state.
    \item Libraries: The shared Git branch is $\mathcal{L}^0_t$. Each agent's local workspace is $\mathcal{L}^\alpha_t$. The inclusion invariant $\mathcal{L}^0_t \subseteq \mathcal{L}^\alpha_t$ holds because copies are initialized from the shared volume and updated via git pull after each public commit.
    \item Resolution and dependencies: A target is resolved when it passes VerifiedAgora's full pipeline. Dependencies $\delta^*(s)$ are extracted post-hoc by the batched axiom collector.
    \item Publication: \texttt{write\_contribution} $\mapsto$ commit to shared branch $\mapsto$ Market Mechanics settlement.
    
    \item Market: \texttt{PriceySatisfiedByAsset(target, stop\_time, price)} implements $\chi^+ = (s, T, \ell)$ with $\ell = \mathrm{price}$. The bounty agent seeds long offers on all open targets at $\ell = 0.1$.
    
    \item Proof kernel: The LLM agent reads the target, inspects the goal state via LSP, writes a tactic proof, and submits via \texttt{write\_contribution}. Lean's type-checker and VerifiedAgora serve as the binary oracle: $(1, \delta^*(s))$ on success, $(0, \bot)$ on failure. The processing time $T^\alpha_s$ is the number of decision epochs consumed by the attempt.
    \item Conjecture kernel: The LLM proposes new intermediate declarations by writing them into the Lean file. If the declaration type-checks and is accepted as a new non-target declaration, it becomes concrete in the agent's private library and may be published. The proposal distribution is induced by the LLM's prior over useful statements conditioned on proof context.
    \item Query model: Given a target, the LLM is prompted to estimate the target's likelihood of success and expected duration to proof, based on its proof history and current library state. These estimates are passed to the strategy module as $(\hat{p}, \hat{\tau})$.
\end{itemize}


\subsection{Synthetic Training Data Generation}\label{sec:training}

Training the strategy module on real Lean is impractical: each proof attempt requires minutes of LLM inference and Lean compilation, and the target set is fixed across episodes. Instead, we train on synthetic MMATPG instances whose structural statistics are calibrated to real Lean repositories, with the LLM subagent replaced by a tractable parameterized kernel. This section describes the calibration pipeline.


\subsubsection{Offline MMATPG Extraction}\label{sec:dep-extraction}


\paragraph{Utility Graph Extraction}\label{sec:util-extraction}

Given a complete target Lean repository, we extract the declaration-level dependency DAG, estimate per-target difficulty, and compute utility weights. Namely, we metaprogrammatically invoke Lean's internal dependency extraction tools, by parsing the built \texttt{.olean} cache, which stores \texttt{Environment}-level data on declarations, their types, values, attributes, and dependencies. This is done recursively over all modules in the target project, excluding infrastructure namespaces (\texttt{Lean}, \texttt{Init}, \texttt{Std}, etc.). The result is a dependency index $\{d \mapsto [d_1, \ldots, d_k]\}$.

From the dependency index, we compute per-declaration dependency frequency (how often a declaration appears as a dependency of others) and relative frequency (normalized by total dependency count). For each target $\phi$, the rarity score is:
\[
\mathrm{rarity}(\phi) = \sum_{p \in \mathrm{deps}(\phi)} \frac{1}{\mathrm{rel\_freq}(p)}.
\]
Targets depending on rare prerequisites receive higher rarity. We quantile-match these scores against a Mathlib-calibrated percentile table (derived from empirical Mathlib dependency statistics) and apply $\log(1+\cdot)$ smoothing to compress outliers.

Normalized rarity maps to difficulty:
\[
d(\phi) = 0.05 + 0.90 \cdot \mathrm{normalized\_rarity}(\phi) \in [0.05, 0.95].
\]


For each ordered pair $(\psi, \phi)$, we compute $k$-hop dependency overlap ($k=2$, decay $0.3$). Direct dependencies: $w(\psi, \phi) = 1.0$. Second-hop overlap: weighted by decay and normalized by target dependency mass. Non-overlapping pairs: floored at $0.01$. The coverage formula is:
\[
w(\psi, \phi) = \mathrm{clip}\!\left(\frac{\sum_x \mathrm{wt}(x) \cdot \min\big(\mathrm{src\_deps}(x),\, \mathrm{tgt\_deps}(x)\big)}{\sum_x \mathrm{wt}(x) \cdot \mathrm{tgt\_deps}(x)},\; 0.01,\; 0.99\right),
\]
overridden to $1.0$ for hard dependencies. This produces a dense utility graph reflecting the structural relationships of the real codebase. This will serve as a distributional prior for sampling structurally similar synthetic instances for training the strategy module.


\paragraph{Kernel Calibration}\label{sec:kernel-calibration}

Unlike at inference-time, during the synthetic data training stage the LLM subagent is replaced by parameterized probability kernels. For example, the synthetic proof kernel from Chapter~\ref{chap:experiments} has parameters $(\rho_0^\alpha, \rho_1^\alpha, \lambda_{\mathrm{diff}}, \kappa)$. We calibrate these heuristically as tunable hyperparameters.

% We note this is a limitation of our evaluation methodology: this arbitrary hyperparameter tuning does not guarantee that the synthetic kernel is a faithful model of the subagent's reasoning capabilities, and thus the strategy module trained on this kernel may learn coordination strategies that are not effective for real LLM agents. Future work could explore more rigorous calibration methodologies, such as collecting detailed performance data from the LLM agents on a representative subset of targets, and fitting the kernel parameters to minimize the discrepancy between the kernel's predicted success rates and the observed success rates across different difficulty tiers and budget levels.

Training proceeds over seven curriculum levels ($n \in \{4, 8, 16, 32, 64, 128, 256\}$ theorem pairs), progressively shifting probability mass toward larger instances. We fix $8$ agents for all instances. Before each rollout, the environment is regenerated, sampling a fresh graph from the calibrated distribution. This ensures the policy sees many structurally similar but non-identical instances, promoting generalization over memorization.

\subsubsection{Training-to-Inference Gap}\label{sec:gap}

The key gap between training and inference is that the parameterized kernels are stationary and Markovian, while the LLM agent is neither: its proof success probability depends on its entire interaction history, and its conjecture proposals depend on mathematical reasoning not captured by the utility-graph heuristic.

We mitigate this through three mechanisms:
\begin{enumerate}
    \item \textbf{Conservative calibration.} Kernel parameters are fitted to observed (not idealized) LLM success rates, so the strategy module does not learn to rely on proof capabilities the LLM does not possess.
    \item \textbf{Kernel noise.} Stochastic noise in the proof/conjecture/query kernels prevents the strategy module from overfitting to deterministic oracle behavior. The trained module must be robust to outcome variance.
    \item \textbf{Prompt-based deployment.} At inference time, the strategy module's recommendations are injected into the LLM's prompt, not enforced as hard constraints. The LLM can deviate from recommendations (and does so on $\sim\!15\%$ of steps), adapting the strategy to its own proof state. This soft coupling absorbs training-to-inference distributional shift.
\end{enumerate}

\subsubsection{Strategy Module Training}\label{sec:strategy-training}

The strategy module is the GPT actor from Chapter~\ref{chap:marl}, trained on PNT-calibrated synthetic instances (\S\ref{sec:training}). Training uses the full MARL pipeline (PPO + GAE + entropy annealing + population-based training + five-level curriculum). The trained module is applied \emph{zero-shot} to the real Lean project.

At deployment, the strategy module observes the Agora scaffold state (embedded as an action-epoch-discretized MMATPG instance) and outputs high-level actions, serialized into the LLM agent's prompt at each decision epoch. The LLM retains full autonomy over proof/conjecture construction.


\subsubsection{Systems Under Comparison}\label{sec:systems}

We compare the following Agora-based neural theorem proving systems:
\begin{itemize}
    \item[\textbf{B1.}] \textbf{Agora Scaffold ($8$ agents).} LLM agents on the Agora platform with no strategy module. The bounty agent seeds offers on all targets. Structural advantages over A2: verified publication, market bounties, standardized MCP interface.
    \item[\textbf{C1.}] \textbf{Agora + Strategy ($8$ agents).} Same scaffold as B1, with the MARL-trained strategy module injecting recommendations into each agent's prompt. In practice, agents follow recommendations on $\sim\!85\%$ of decision epochs.
\end{itemize}

Against the following agentic LLM baselines:
\begin{itemize}
    \item[\textbf{A1.}] \textbf{Claude Code ($1$ agent).} A single Claude instance with Lean LSP access. Serial processing, no coordination.
    \item[\textbf{A2.}] \textbf{Claude Code Teams ($8$ agents).} A single Claude Code Teams instance with Lean LSP access specified to have 8 parallel agents.
    \item[\textbf{A3.}] \textbf{Aristotle Agent.} State-of-the-art agentic Lean prover, evaluated as a black box.
    \item[\textbf{A4.}] \textbf{OpenGauss.} Another SOTA system, same evaluation protocol.
\end{itemize}


\begin{remark}[Strategy module as controller]
The action-space mapping is: $\mathrm{Prove}(\phi) \mapsto$ direct the LLM to attempt target $\phi$; $\mathrm{Pub}(\phi) \mapsto$ call \texttt{write\_contribution}; $\mathrm{Qry}(\phi) \mapsto$ prompt for difficulty estimate; market actions $\mapsto$ Agora market MCP tools. The LLM may deviate because the strategy module's output is a prompt injection, not a hard constraint.
\end{remark}

\begin{remark}[LLM subroutines]
Future work may study a variant where the strategy module triggers separate LLM subroutines with specialized prompts (e.g., a ``proof attempt'' subroutine), more closely mirroring the theoretical MMAPTG separation of strategy and proof kernel.
\end{remark}


\subsubsection{Evaluation Protocol}\label{sec:protocol}

All systems receive the same fixed total wall-clock time budget $B$. Moreover, we focus on the following metrics:
\begin{itemize}
    \item $\mathrm{Resolve\%}$: fraction of $\Phi^*$ with sorry-free, VerifiedAgora-certified proofs (overall and by tier).
    \item $\mathrm{WallTime}_{x\%}$: wall-clock time to reach $x\%$ resolution.
    \item $\mathrm{Duplication\%}$: fraction of proof attempts targeting a declaration already solved by another agent.
    \item $\mathrm{DepUnlock}$: count of dependency-unlock events.
    \item $\mathrm{Discovery}$: number of new (non-target) declarations introduced and used in downstream proofs.
\end{itemize}

The core evaluation is the standard sorry-filling task: given the full PNT repository with all targets (MiniCTX-v2) replaced by sorry (and equally economically bountied, as necessary), systems must resolve as many targets as possible within the time budget. This tests the overall effectiveness of the coordination mechanisms in guiding agents to efficiently allocate their proof efforts.

\subsubsection{Limitations}\label{sec:limitations}

\begin{itemize}
    \item The arbitrary kernel hyperparameter tuning does not guarantee that the synthetic kernel is a faithful model of the subagent’s reasoning capabilities, and thus the strategy module trained on this kernel may learn coordination strategies that are not effective for real LLM agents. Future work could explore more rigorous calibration methodologies, such as collecting detailed performance data from the LLM agents on a representative subset of targets, and fitting the kernel parameters to minimize the discrepancy between the kernel’s predicted success rates and the observed success rates across different difficulty tiers and budget levels.
    \item The strategy module was trained on synthetic instances and applied zero-shot. Domain shift may reduce effectiveness; fine-tuning on real Lean data is a natural next step.
    \item Comparisons to A3/A4 are against public configurations not specifically tuned for MiniCTX PNT.
    \item Compute normalization is approximate: LLM API calls, Lean kernel checks, and strategy-module inference have different costs. Decision-epoch discretization (\S\ref{sec:time-discretization}) is a simplification of the true asynchronous setting.
    \item The Agora market is simplified relative to full MMATPG theory: contract types are restricted to target declarations, deadlines are coarsened, and the bounty agent uses fixed prices.
\end{itemize}

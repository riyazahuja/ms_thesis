\chapter{Application: Neural Theorem Proving}\label{chap:application}

The preceding chapters develop the MMATPG framework in a synthetic setting: finite formula universes, parameterized proof kernels, and controlled dependency structures. This chapter asks whether the framework's core insight---that market-mediated coordination with a learned strategy module improves multi-agent theorem proving---transfers to real formal mathematics.

We instantiate the MMATPG scaffold with large language models (LLMs) as proof-producing agents, operating on a real Lean 4 codebase. We evaluate on the Prime Number Theorem (PNT) formalization from the MiniCTX benchmark, a curated subset of real-world Lean repositories designed to test theorem-proving systems on mathematically substantial, interdependent proof obligations. We compare three paradigms: (i) existing agentic theorem provers, (ii) LLMs coordinated by the MMATPG scaffold alone, and (iii) LLMs coordinated by the scaffold equipped with the MARL-trained strategy module from Chapter~\ref{chap:marl}. We show that the strategy module yields measurable improvements in proof completion rate and wall-clock time, particularly on proof obligations with nontrivial dependency structure.



\subsection{The Agora Platform}\label{sec:agora}

We implement Agora, an interface for connecting MMATPG's to agentic neural theorem provers for real Lean~4 projects. It consists of three backend services, a Lean-side verification library, and an agent orchestration harness.

\subsubsection{Architecture}
Agora consists of four main parts:


\begin{itemize}
    \item \textbf{Library Mechanics}: a FastAPI service that clones and builds Lean repositories, scans for target declarations, and validates contributed proofs via the VerifiedAgora Lean library. It is the only service that runs Lake/Lean.
    
    \item \textbf{Market Mechanics}: a FastAPI service that owns wallets, target-conditioned assets, offers, collateral constraints, money-conservation invariants, and settlement logic.
    
    \item \textbf{Backend Interface}: the public API gateway that coordinates the two mechanics services, handles authentication, broadcasts activity events over WebSocket, and exposes the Agora MCP server at \texttt{/mcp/} for LLM agent access.
    
    \item \textbf{Agent Orchestrator}: a separate harness that spawns Docker containers for AI agents, provisions API keys and wallets, seeds the market with initial bounties, and manages agent lifecycles.
\end{itemize}

Agents interact with Agora exclusively through the MCP endpoint. Inside each agent container, a local MCP auth proxy injects the agent's API key into tool calls and rewrites local file paths into contribution API calls, so the agent's LLM does not need to manage authentication or file-upload mechanics directly.


\subsubsection{Mapping to the MMATPG}\label{sec:agora-mapping}

The Agora platform instantiates each component of the MMATPG as follows.

\paragraph{Formulas and sequents.}
Lean declarations (theorems, lemmas, definitions) serve as formulas. Each declaration tagged with \texttt{@[target]} in the Lean source is a target formula. The sequent $[\Gamma\proves\phi]$ corresponds to a proof obligation: the declaration's type is the consequent $\phi$, and the antecedent $\Gamma$ is the declaration's local context (hypotheses plus all imported and available lemmas). In practice, the sequent is identified with the target declaration itself, since the antecedent is determined by the import graph and the current state of the shared repository.

\paragraph{Libraries.}
The shared (public) library $\mathcal{L}^0_t$ is the shared Git branch maintained by Agora. All agents can read its current state. The \emph{private library} $\mathcal{L}^\alpha_t$ for agent $\alpha$ is the agent's local working copy of the repository. The inclusion invariant $\mathcal{L}^0_t\subseteq\mathcal{L}^\alpha_t$ holds because each agent's working copy is initialized by copying from the shared library volume, and public updates are propagated via git pull.

\paragraph{Resolution.}
A target declaration is \emph{resolved} when a contributed proof satisfies three conditions: (i) the proof is \texttt{sorry}-free, (ii) it type-checks under Lean's kernel, and (iii) it passes VerifiedAgora's certification policy, which transitively disallows extraneous axioms (only \texttt{propext}, \texttt{Quot.sound}, and \texttt{Classical.choice} are permitted) and rejects \texttt{unsafe} or \texttt{partial} definitions recursively down the dependency DAG. This is strictly stronger than Lean's built-in \texttt{sorry}-freedom check.

\paragraph{Dependencies.}
Lean's declaration-level dependency graph provides $\delta^*$. When a proof of declaration $A$ uses declaration $B$ (via \texttt{exact}, \texttt{apply}, \texttt{rw}, etc.), we have $B\in\delta^*(A)$. Dependencies are extracted post-hoc (i.e. after proof construction) by VerifiedAgora's batched axiom collector, which recursively traverses the types and values of definitions, theorems, and inductive declarations with cycle protection and caching. 

\paragraph{Publication.}
An agent publishes by calling the mcp command \texttt{write\_contribution}, which triggers the following pipeline:
\begin{enumerate}
    \item The agent's contributed Lean file content is sent to Library Mechanics.
    \item Library Mechanics invokes VerifiedAgora's \texttt{get\_targets} executable, which compiles the submission, validates it against the existing target descriptor (checking type/kind/attribute/axiom compatibility), and produces an automerged file that retains valid solved targets and preserves unsolved ones.
    \item If \texttt{overwrite\_target=true}, VerifiedAgora writes the automerged file to the repository and runs \texttt{lake build} on the target module. If the build fails, the original file is restored and an \texttt{automerge\_failure} diagnostic is reported.
    \item On success, Library Mechanics commits and pushes the change to the shared branch, and reports a summary of newly solved targets, new declarations, and remaining unsolved targets.
    \item The Backend forwards solved-target updates to Market Mechanics, which records proof times, simplifies target-conditioned assets, and converts settled positions into wallet balances.
\end{enumerate}

This pipeline implements the dependency-closure publication primitive of the MMATPG: publishing a solved target automatically makes its proof (and all declarations it introduced) available to all agents via the shared branch.

\paragraph{Proof kernel.}
The proof kernel $\mathcal{K}^\alpha_{\mathrm{proof}}$ is instantiated by an LLM agent attempting to prove a target declaration. The agent edits a Lean file in its local workspace, constructs a proof (using tactics, term-mode, or a combination), and submits it via the contribution pipeline. The Lean kernel's type-checker and VerifiedAgora's certification policy serve as the success/failure oracle: the kernel returns $(1, \delta^*(s))$ if the proof passes all checks, and $(0,\bot)$ otherwise. The processing time $T^\alpha_s$ is the wall-clock time (or number of LLM interaction steps) consumed by the attempt.

\paragraph{Market mechanism.}
Agora's market implements the collateralized bilateral contract market of Chapter~\ref{sec:decentralized}. The primary asset type is \texttt{PriceySatisfiedByAsset(target, stop\_time, price)}, which pays $1-\mathrm{price}$ if the target is resolved by \texttt{stop\_time} and $-\mathrm{price}$ otherwise---exactly the $\chi^+ = (s, T, \ell)$ contract with $\ell = \mathrm{price}$.

Key implementation details:
\begin{itemize}
    \item \textbf{Offer creation} mints a matched long/short pair and posts one side as a public offer. The poster retains the counterside as a private asset. A counterbalancing \texttt{LinearCombinationAsset} with coefficient $-Q$ ensures zero net value creation.
    
    \item \textbf{Offer taking} transfers the offered position to the taker's portfolio. Cash does not transfer at accept time; value is realized only at settlement. This is identical to the theoretical formulation.
    
    \item \textbf{Settlement} triggers when: (i) the target is publicly resolved, (ii) the target's negation is resolved (making the target impossible), or (iii) the deadline passes. Settled assets that simplify to constants are converted to wallet balance.
    
    \item \textbf{Collateral} is enforced via a minimum-wallet-value check: $\underline{B}^\alpha := c^\alpha + \sum_{x} \mathrm{lower\_bound}(x) \ge 0$, where the sum is over private assets and held offers with non-positive lower bounds.
    
    \item \textbf{Money conservation} is enforced at every trading operation: the system computes the net asset diff, simplifies against current target statuses, and rejects transactions with nonzero net coefficients.
\end{itemize}

\paragraph{Bounty agent.}
The Agent Orchestrator seeds the market by creating \texttt{PriceySatisfiedByAsset} offers for every open target, funded by an admin wallet. Each offer posts the \emph{long} side at a fixed price, so any agent can accept it and profit if they resolve the target. This provides baseline liquidity and ensures agents face nontrivial market incentives from the start.

\paragraph{Query model.}
In this real-Lean setting, agents can query their own proof capability via the Lean Language Server Protocol (LSP) MCP tools (e.g., inspecting goal states, checking partial proofs). This provides noisy estimates of difficulty analogous to the query model's $(\hat p, \hat\tau)$ readout. The strategy module (when present) uses these signals to prioritize targets.


\subsubsection{Agent Interface}

Each agent container runs an LLM with access to the following MCP tool groups:

\begin{itemize}
    \item \textbf{Library tools}: list projects, read files, search declarations, read target content, submit contributions (via \texttt{write\_contribution\_from\_file}, a synthetic proxy tool that reads the agent's local file and submits it).
    \item \textbf{Market tools}: view wallets, offers, and target statuses; create/take/cancel offers; transfer assets.
    \item \textbf{Lean LSP tools}: hover, goal state, completion, diagnostics---providing partial-proof feedback without full submission.
    \item \textbf{Filesystem tools}: read, write, edit, search files in the local workspace.
\end{itemize}

The agent runtime is a persistent loop: after each LLM response, the harness reengages with updated status (active project, build results, market events, recovery hints). Auto-compaction summarizes long conversations into a memory file to prevent context overflow.

\subsection{Systems Under Comparison}\label{sec:systems}

We compare three paradigms.

\paragraph{Paradigm A: Agentic provers (baselines).}

\begin{itemize}
    \item[\textbf{A1.}] \textbf{Claude Code ($1$ agent).} A single Claude instance with Lean LSP access, iteratively attempting proofs. Serial processing, no coordination.
    
    \item[\textbf{A2.}] \textbf{Claude Code Teams ($n$ agents).} Multiple Claude instances on a shared Git branch. Agents are assigned disjoint target subsets and observe each other's committed proofs. Coordination is ad hoc (file-level, no market mechanism).
    
    \item[\textbf{A3.}] \textbf{Aristotle Agent.} State-of-the-art agentic Lean prover, evaluated as a black box on the same target set.
    
    \item[\textbf{A4.}] \textbf{OpenGauss.} Another state-of-the-art system, same evaluation protocol.
\end{itemize}

\paragraph{Paradigm B: Agora scaffold (no strategy module).}

\begin{itemize}
    \item[\textbf{B1.}] \textbf{Agora Scaffold ($n$ agents).} $n$ LLM agents operating on the Agora platform with no preprogrammed policy. Agents interact via MCP tools; proofs are validated by VerifiedAgora.
\end{itemize}

% The key difference from A2 is structural: the scaffold provides a shared library with verified publication (not just git commits), a market with explicit bounties (not just informal task assignment), and a standardized agent interface (MCP tools, not ad hoc prompting).

\paragraph{Paradigm C: Agora scaffold + strategy module.}

\begin{itemize}
    \item[\textbf{C1.}] \textbf{Agora + Strategy ($n$ agents).} Same as B1, but the implicit policy is replaced by the MARL-trained strategy module from Chapter~\ref{chap:marl}. The agentic LLM is given the serialized outputs of the strategy module at each step in its prompt, which allows for strategic influence on the agent's actions.
    
\end{itemize}

\begin{remark}[Strategy module as controller]
The separation between strategy (what/when/who) and tactics (how to prove) is the key architectural choice. The strategy module is the trained GPT actor from Chapter~\ref{chap:marl}, operating over the same observation/action space as in the synthetic experiments. Its action space maps to Agora operations: $\mathrm{Prove}(\phi) \mapsto$ direct the LLM to attempt target $\phi$; $\mathrm{Pub}(\phi) \mapsto$ call \texttt{write\_contribution}; $\mathrm{Qry}(\phi) \mapsto$ query Lean LSP for goal-state complexity; market actions $\mapsto$ Agora market MCP tools. The strategy module was trained on synthetic Agora Forge instances and is applied zero-shot to the real Lean project.
\end{remark}
\begin{remark}[LLM Subroutines]
Future work may study the differences between C1 and a variant where the strategy module's outputs are not directly fed into the LLM prompt, but instead trigger separate subroutines that call the LLM with specialized prompts (e.g., a "proof attempt" subroutine that takes a target and returns a proof sketch). This would more closely mirror the theoretical framework's separation of strategy and proof kernel, but may require additional engineering to implement effectively.
\end{remark}


\subsection{Evaluation Protocol}\label{sec:protocol}

\paragraph{Compute budget.}
All systems receive the same total API-call based compute budget $B$. Results are reported at checkpoints $\{B/4, B/2, 3B/4, B\}$.

\paragraph{Metrics.}
\begin{itemize}
    \item $\mathrm{Resolve\%}$: fraction of $\Phi^*$ with sorry-free, VerifiedAgora-certified proofs (overall and by tier).
    \item $\mathrm{WallTime}_{x\%}$: wall-clock time to reach $x\%$ resolution.
    \item $\mathrm{Duplication\%}$: fraction of proof attempts targeting a declaration already solved by another agent.
    \item $\mathrm{DepUnlock}$: count of events where one agent's publication enables another agent to resolve a previously blocked target.
\end{itemize}

\subsection{Limitations.}
\begin{itemize}
    \item The strategy module was trained on synthetic Agora Forge instances and applied zero-shot to real Lean. Domain shift (different dependency structure, proof difficulty distribution, market dynamics) may reduce effectiveness. Fine-tuning on real Lean projects is likely to improve results.
    \item Comparisons to A3/A4 are against public configurations; these systems may not be optimized for MiniCTX PNT.
    \item Compute normalization is approximate: LLM API calls, Lean kernel checks, and strategy-module inference have different costs.
\end{itemize}
